# ADR 0004：简体权威与构建期派生繁体

- 状态：已采纳（Accepted）
- 日期：2026-07-09
- 关联 issue：#34
- 修订：ADR 0003 第 2 条中「LanguageToggle 占位保持不实现」的临时条款

## 背景

站点正文已按中华书局点校本《古诗源》简体 epub 重导入，简体 Markdown 是唯一权威内容源。繁体读者仍希望用传统字形阅读全站，但维护两套内容文件会导致正文、目录、注解与搜索索引漂移。

ADR 0003 已把未来方向定为「简体权威、派生繁体」。#34 将该方向落地为全站简繁切换。

## 决策

1. **简体 Markdown 继续作为唯一权威源。** 不新增繁体 Markdown，不新增独立繁体路由；页面 metadata 保持简体权威。

2. **繁体文本在构建期派生。** 使用 OpenCC 通用简转繁（s2t）词典，把首页目录、卷页、作者页、诗页正文、源流注解、搜索索引和固定 UI 文案派生为繁体，并与简体文本一同传给客户端。

3. **客户端切换只选择显示变体。** 简繁偏好使用 `gushiyuan-script-variant` 写入 localStorage，无偏好时默认繁体；切换时同步更新 `<html lang>` 为 `zh-CN` / `zh-Hant`。页面 metadata / OG locale 仍保持简体权威（`zh_CN`）。

4. **一对多转换用仓库内覆盖表修正。** `content/script-conversion-overrides.json` 记录古文语境的整词/短语覆盖，例如「诗云」保持「詩云」、「千里」保持「千里」。发现误转时补覆盖表和测试。

5. **字体子集收集繁体派生码点。** `collectSiteFontGlyphs` 在扫描内容和 UI 文案时同时纳入派生繁体码点，子集 cmap 校验继续作为缺字防线。

## 后果

- 正面：全站可在同一 URL 空间中切换简繁，内容仍只有一份权威源；搜索可用繁体输入命中简体内容。
- 负面：页面会内嵌简繁两份文本，静态 HTML/客户端数据体积约随文本量增加一倍；转换质量长尾依赖覆盖表持续维护。
- ADR 0003 的「简体优先」仍成立；本 ADR 只取代其 LanguageToggle 暂不实现的临时结论。
